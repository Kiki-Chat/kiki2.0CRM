## TL;DR — what you need to do

1. **Google Cloud Console**: create an OAuth 2.0 Client ID for "HeyKiki Portal" with the right scopes and redirect URIs (~10 min).
2. **Azure**: register an app with Microsoft Graph permissions (~10 min).
3. **Paste 4 credentials** into `backend/.env` (local dev) and Railway env vars (prod) — see §3.
4. Restart backend. **Done.**

## 1. Google Cloud Console — Gmail + Calendar

### 1.1 Create / select project
- Go to <https://console.cloud.google.com/>
- Create a new project named `heykiki-portal` (or reuse an existing one if you have one).
- Note the project ID.

### 1.2 Enable APIs
APIs & Services → Library → enable each:
- **Gmail API**
- **Google Calendar API**

### 1.3 OAuth consent screen
APIs & Services → OAuth consent screen:
- **User type:** External (so non-google-workspace customer admins can also log in)
- **App name:** `HeyKiki Portal`
- **User support email:** `support@heykiki.de`
- **Developer contact:** your email
- **App domain:** `heykiki.de`
- **Authorized domains:** `heykiki.de`, `up.railway.app`
- **Scopes** (click Add or Remove Scopes — search and select these exact ones):
  - `https://www.googleapis.com/auth/gmail.send` — Send-only Gmail (we don't read inboxes; least-privilege)
  - `https://www.googleapis.com/auth/calendar` — Full read+write to Calendar (needed for P1.9 two-way sync)
  - `openid`, `email`, `profile` — Standard sign-in claims
- **Test users** (while app is in "Testing" mode): add `agrawalamber01@gmail.com` and any other dev emails. Once verified, switch to "In production" so any customer admin can authorize.

### 1.4 OAuth client credentials
APIs & Services → Credentials → Create credentials → OAuth client ID:
- **Application type:** Web application
- **Name:** `HeyKiki Portal Web`
- **Authorized JavaScript origins:**
  - `http://localhost:5173` (frontend dev)
  - `https://frontend-production-4bdf.up.railway.app` (frontend prod)
- **Authorized redirect URIs** (these MUST exactly match what the backend sends):
  - `http://localhost:8000/api/settings/oauth/google/callback`
  - `https://backend-production-3f88a.up.railway.app/api/settings/oauth/google/callback`

→ Click **Create**. You get a **Client ID** (`…apps.googleusercontent.com`) and **Client secret** (`GOCSPX-…`). Note both.

## 2. Azure — Microsoft Graph (Outlook Mail + Calendar)

### 2.1 App registration
- Go to <https://portal.azure.com/> → Microsoft Entra ID → App registrations → New registration.
- **Name:** `HeyKiki Portal`
- **Supported account types:** "Accounts in any organizational directory and personal Microsoft accounts" (so both Microsoft 365 customers and personal Outlook.com users can authorize).
- **Redirect URI:**
  - Platform: Web
  - URI: `https://backend-production-3f88a.up.railway.app/api/settings/oauth/microsoft/callback`

→ Register. You land on the app's overview. Note the **Application (client) ID**.

### 2.2 Add the localhost dev redirect URI
- In the registered app → Authentication → Add a platform → Web:
  - `http://localhost:8000/api/settings/oauth/microsoft/callback`
- Under "Implicit grant and hybrid flows", leave both checkboxes UNCHECKED (we use auth-code flow with PKCE).

### 2.3 API permissions
- API permissions → Add a permission → Microsoft Graph → **Delegated permissions**:
  - `Mail.Send`
  - `Calendars.ReadWrite`
  - `User.Read` (standard "who am I" — added by default; keep it)
  - `offline_access` (so we get a refresh token)
- Click "Grant admin consent for HeyKiki Portal" if you have admin rights on a directory (optional but cleaner for org-managed accounts).

### 2.4 Client secret
- Certificates & secrets → New client secret → Description `HeyKiki Portal`, Expires 24 months (max).
- **Copy the secret VALUE immediately** (Azure hides it after you leave the page). Note it.

## 3. Paste credentials into env

### 3.1 Local dev — `backend/.env`
Append these four lines (replace each `…` with the real value):

```
GOOGLE_CLIENT_ID=…apps.googleusercontent.com
GOOGLE_CLIENT_SECRET=GOCSPX-…
MS_CLIENT_ID=…
MS_CLIENT_SECRET=…
```

Also set the backend's public URL (used to build redirect URIs at runtime):

```
BACKEND_PUBLIC_URL=http://localhost:8000
```

For prod, set it to `https://backend-production-3f88a.up.railway.app` instead.

### 3.2 Railway — same 5 vars
Railway dashboard → backend service → Variables → bulk paste:

```
GOOGLE_CLIENT_ID=<same as local>
GOOGLE_CLIENT_SECRET=<same as local>
MS_CLIENT_ID=<same as local>
MS_CLIENT_SECRET=<same as local>
BACKEND_PUBLIC_URL=https://backend-production-3f88a.up.railway.app
```

Or via CLI:
```bash
railway variables --set "GOOGLE_CLIENT_ID=…" --set "GOOGLE_CLIENT_SECRET=…" \
                  --set "MS_CLIENT_ID=…"     --set "MS_CLIENT_SECRET=…" \
                  --set "BACKEND_PUBLIC_URL=https://backend-production-3f88a.up.railway.app" \
                  --service backend
```

### 3.3 Restart
- Local: `pkill -f "uvicorn app.main:app"; cd backend && ./.venv/bin/uvicorn app.main:app --port 8000` — see [[backend_no_reload]] memory; restart required.
- Railway: env-var change auto-redeploys. Wait ~30s for the new deploy.

### 3.4 Smoke-test
```bash
curl -H "Authorization: Bearer $JWT" \
     "https://backend-production-3f88a.up.railway.app/api/settings/oauth/google/authorize"
# → expect 200 with {"url": "https://accounts.google.com/o/oauth2/v2/auth?..."}
```

If you get `503 OAuth not configured`, the env vars aren't loaded — recheck the restart.

## 4. What the backend does next (already shipped — Phase 2)

Once the credentials above are loaded:

- `GET  /api/settings/oauth/{provider}/authorize` — returns the OAuth authorize URL with a CSRF state token. Frontend opens it in a popup.
- `GET  /api/settings/oauth/{provider}/callback` — handles the redirect from Google / Microsoft, exchanges the auth code for tokens, encrypts them with the existing `SETTINGS_ENC_KEY` (Fernet) and stores them on `email_configs`, then closes the popup.
- `POST /api/settings/oauth/{provider}/disconnect` — clears the stored tokens.

Migration `0019_oauth_email_configs.sql` adds the columns:
- `oauth_provider text` (`'google'` | `'microsoft'` | `NULL`)
- `oauth_refresh_token_encrypted text`
- `oauth_access_token_encrypted text`
- `oauth_token_expires_at timestamptz`
- `oauth_account_email text` (the email we're authorized to send AS)

The Gmail-send / MS-Graph-send refactor (using OAuth tokens instead of SMTP) lands in Phase 3 alongside the frontend buttons.
