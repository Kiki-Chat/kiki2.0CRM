## ✅ 1. Aktionen / Call-Log rows routed everything to the customer card
**Fixed.** [`CallLogsPage.tsx`](frontend/src/pages/CallLogsPage.tsx) + [`actions.py`](backend/app/api/routes/actions.py)

Root cause: the `onSelect` handler special-cased only `alt_time_proposal`; **every
other** action kind fell through to `navigate(/customers/:id)`. So KVA actions,
appointment confirmations and callbacks all dumped you on the customer card with no
way to act.

Fix — each kind now routes to where it can actually be actioned:
| Kind | Destination |
|---|---|
| `kva_to_send`, `kva_pending_acceptance` | `/cost-estimates/:id` (the KVA itself) |
| `termin_anfrage`, `alt_time_proposal` | opens the related **call** (its inline card has Bestätigen/Ablehnen) |
| `callback_owed` | customer card (has the phone to call back) |
| any of the above with no resolvable call | customer card (graceful fallback) |

Backend: `termin_anfrage` now resolves a `call_id` (via the inquiry's `call_id`, else
the booking conversation) — it was hard-coded `None`, which is why these always fell
back. Also fixed a latent bug in `_alt_time_proposal`: it queried the non-existent
column `calls.conversation_id` (real column = `elevenlabs_conversation_id`) — would
have 500'd the whole endpoint the first time an alt-proposal carried a conversation id.

## ⚠️ 3a. "Customer detail crashes on edit / first load" — could NOT reproduce
Tested live on `main` with this org's data: customer **list**, customer **detail**,
**Bearbeiten** (open) and **Aktualisieren** (save) all work — no crash.

The on-screen message the user saw ("this page cannot be reloaded") is the
[`ChunkErrorBoundary`](frontend/src/components/ChunkErrorBoundary.tsx) fallback
("Diese Seite konnte nicht geladen werden" + "Neu laden"). It fires on **lazy-chunk
load failures**, and only auto-reloads for chunk errors. The most likely real cause is
**deploy/chunk skew**: after a Railway redeploy, a still-open tab holds the old
`index.html` referencing hashed chunks that no longer exist → navigating to a lazy
route (customers, etc.) 404s the chunk → the error screen, and a plain reload doesn't
fix it if the stale `index.html` is still cached.

**Needs:** confirm whether the crash was on the **deployed** app right after a deploy
(very likely) vs. a fresh load. The robust fix is a cache policy — serve `index.html`
`no-cache` / `must-revalidate` and the hashed `/assets/*` `immutable` — so a returning
tab always re-fetches the current `index.html`. (Frontend is served by `serve -s dist`;
a `serve.json` can set these headers.) Not changed yet — pending confirmation of the
environment, since the component code itself is sound.

## 🔎 6. "Toilet emergency" not flagged as emergency in the call log — CONFIRMED bug
Verified with the actual call (`0516d1f7`, "Toilet Emergency Inquiry", Thomas Muller):
- The AI summary itself says *"Es handelt sich um eine Notfallsituation… außerhalb der
  regulären Geschäftszeiten… Das Anliegen wird als dringend markiert"*, and the agent
  offered the Notdienst transfer (customer declined).
- Yet `inquiries.type = "info"` and `emergency_flag = **false**`.

**Root cause** — [`inquiries.py:78`](backend/app/services/inquiries.py#L78):
```python
emergency = outside_hours and agent_urgent
```
`agent_urgent` is only true when `call.data_collection` contains a truthy
`is_emergency`/`emergency`/`notfall`/`urgent` key. **The agent's data-collection schema
never emits such a field** (this call's `data_collection` has none), so `agent_urgent`
is essentially always false → **emergencies are never flagged**, regardless of language.
The German-keyword theory is secondary: even a German emergency call fails the same way.
The agent's *in-call* emergency behavior (prompt-driven) and the *log* flag (this
data_collection gate) are two independent signals that have diverged.

**Proposed fix (needs your OK — it changes which calls get flagged):** make the log flag
derive emergency from call **content**, language-agnostically, instead of a field the
agent never sets. Concretely, when `outside_hours` is true, also treat it as urgent if
the summary/transcript (`summary_title`, `ultimate_summary`, `issue_summary`,
`next_action`) matches a **bilingual** emergency set (Notfall/Notdienst/dringend/
Rohrbruch/Gasgeruch + emergency/urgent/flooding/leak/burst/gas…), and/or capture
`hk_transferCall(emergency=true)` when a transfer does happen. Single source of truth so
the badge matches the agent's behavior. **Not changed yet** — pending your decision on
the approach (content heuristic vs. a real `emergency` data-collection field on the agent).
