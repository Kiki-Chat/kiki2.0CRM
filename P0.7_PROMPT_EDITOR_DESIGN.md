## 1. Goals & non-goals

**Goals**
1. Customers edit **only** the fields they own (brand name, persona tone, opening greeting, services, special instructions, FAQ, address). System flow (tool calls, identify-customer scaffolding, booking sequence, system-variable placeholders, webhook URLs) is invisible to them and unbreakable.
2. The exact regression that triggered this P0 — a customer-editable rule (`CATEGORY CHECK: book only maintenance`) buried in a 50k-char system prompt blocking *all* repair bookings — is **structurally impossible** under the new model: there is no place in the form where a customer can insert flow-control logic.
3. The agent's final prompt is **deterministically composed** from a versioned master template + the customer's structured fields. Same inputs → same prompt, every time.
4. **Super-admin escape hatch:** the raw prompt + master-template view stays available to super-admins for debugging.
5. **Migration:** existing customers (today only `kiki-test-007` is real) move to the new model cleanly. Bespoke edits in their current raw prompt are surfaced for explicit reconciliation, not silently dropped.

**Non-goals**
- Free-form prompt editing for customers (super-admin only).
- Per-section rollback (Verlauf & Rückgängig already covers prompt-level rollback via snapshots).
- A WYSIWYG / no-code prompt builder beyond the field set listed in §3.
- Multi-language prompts. Per Amber: **DE-formal across the board for all real customers**; existing English `first_message` strings only persist for legacy reasons.
- Renaming or restructuring tools / `client_events` — those are the safety layer's domain, not the editor's.

|---|---|---|

## 4. Composition logic

### 4.1 Master template (lives in `backend/app/services/prompt_composer.py`)

A single Python string with `{{field_name}}` placeholders. Composed by `compose_prompt(org_id) -> str` which:
1. Loads org row, agent_configs row, business_hours, categories, FAQ, address, management.
2. Renders each field through an **escape pass** that strips `{{...}}`, `{system__*}`, and any `Du darfst NUR` / `MUST` / `MANDATORY` directives a customer might paste — preventing prompt-injection.
3. String-replaces placeholders in the master template.
4. Returns the final string. Deterministic — same inputs always produce identical output.

**Sketch of the template** (abridged — full version in code):

```text
Du bist Kiki, die {{persona_tone_descriptor}} Telefon-Assistentin von
{{brand_name}}. {{trade_specialty}}

Du nimmst Anrufe für {{brand_name}} entgegen. Anrufer können bestehende oder
neue Kunden sein. Du hast Zugriff auf CRM-Tools für Terminbuchung, Kundendaten,
Anliegen, Projekte und Kostenvoranschläge. Die aktuelle Zeit in Deutschland ist
{{system__time}}.

### Erkennung & Verarbeitung
[… SYSTEM-MANAGED: Time-of-Day Mode, Existing Customer Reference, Step 0,
Step 1, Step 2, Step 3 booking flow including the new universal scope …]

### Schritt 3 — Termin
{{appointment_scope}}

Der unten beschriebene Buchungsablauf gilt für alle Anliegen.
[… SYSTEM-MANAGED: ask verbatim, narrow down, two suggestions, etc. …]

### Wissensbasis
**Unternehmen**
{{brand_name}} — {{trade_specialty}}.
{{full_address}}.
{{management_block}}

**Geschäftszeiten**
{{business_hours_block}}
Mittagspause: {{lunch_break}}

**Leistungsangebot**
{{services_offered}}

**Leistungen, die wir nicht anbieten**
{{services_not_offered_block}}

**Häufige Fragen**
{{faq_block}}

**Zusätzliche Hinweise**
{{knowledge_text}}

{{special_instructions}}

### Notfall
[… SYSTEM-MANAGED + emergency_surcharge_text injection …]
```

The full template is **~3000 chars** of skeleton + customer-field placeholders. The composed prompt will land in the same **~30–50k char range** as today's prompt once all fields are populated — but every long stretch is either template (system-managed) or a customer-controlled field with strict length caps.

## 5. Migration plan for existing customers

Currently only **one** real customer-bound agent exists: `agent_5001ksahz3w7fhx90j71xr800py4` for `kiki-test-007`. Its current prompt has been hand-edited (Husmann/Dreier rebrand, CATEGORY CHECK insertion, etc.). When P0.7-build ships, that prompt must move to composer-driven without losing customer intent.

**Strategy: manual migration with a parser-assisted assist, super-admin-only.** Auto-parsing 51k chars of free-form English/German prose into structured fields with confidence is not realistic.

**Steps (run by super-admin per org, NOT by org_admin):**
1. **Snapshot the current raw prompt** (already happens — every Kiki-Zentrale write snapshots).
2. **Run `prompt_composer.compose_prompt(org_id)` against currently-populated agent_configs/org fields** to get the "what the composer would produce today" output.
3. **Open the two side-by-side in the super-admin migration view** (NEW UI, lives behind `/super-admin/orgs/{id}/prompt-migration`):
   - Left: current raw prompt (read-only).
   - Right: composer output (read-only).
   - Bottom: form with the new structured fields, pre-filled from current `agent_configs` + parser hints (regex-extracted brand name, address, etc.).
4. **Super-admin tunes the form fields** until composer output is acceptable.
5. **Commit:** `PATCH /api/super-admin/orgs/{id}/persona-fields` saves the form values + immediately calls `compose_prompt` + calls `patch_agent_safely(endpoint_label="prompt-migration")`. The old snapshot stays in `agent_config_snapshots` for rollback.
6. From this point on, the org_admin sees the new structured editor; the raw textarea is gated behind `super_admin`.

**Grandfathering rule:** if an org has `agent_configs.uses_composed_prompt = TRUE` (a NEW boolean column added in the migration), the structured editor is shown. If `FALSE` or NULL, the raw editor is shown (legacy mode) with a banner "Diese Organisation nutzt den alten Editor. Bitte HeyKiki Support kontaktieren für die Migration." This means **no customer is forced to migrate**; super-admin migrates each one on a schedule.

## 6. Super-admin escape hatch

The new structured editor REPLACES the raw textarea for `org_admin`. For `super_admin`:
- A toggle `"⚡ Erweiterter Modus"` in the editor toolbar reveals the raw `prompt.prompt` (read-only by default, edit unlocked via a confirmation modal).
- A separate `"Master-Template ansehen"` link shows the composer template + the field substitution map, for debugging.
- Verlauf (history) shows BOTH composer-driven writes (`endpoint_label="persona-fields"`) and raw writes (`endpoint_label="prompt-editor"` or `"prompt-migration"`).
- Direct raw edits by super-admin set `agent_configs.uses_composed_prompt = FALSE` until super-admin re-migrates — so the customer doesn't see a stale structured form that no longer matches the actual deployed prompt.

## 7. Preview flow

In the structured editor, a sticky bottom-bar button **"Vorschau"** triggers `POST /api/kiki-zentrale/persona-fields/preview` which:
1. Takes the **unsaved** form values as input.
2. Runs `compose_prompt(org_id, override_fields=<form_values>)` server-side.
3. Returns the composed prompt as a **read-only string**.

The UI shows it in a modal / right-pane with:
- Header: "So wird Kiki Ihren Prompt sehen"
- The composed prompt in a monospace `<pre>`, syntax-highlighted only at field-boundary level (no edit affordances).
- A diff toggle: "Im Vergleich zur aktuellen Version" — uses the existing `POST /api/kiki-zentrale/prompt/diff` endpoint.
- "Schließen" button.

Customer cannot paste anything from the preview back into a raw input. The only way for a customer to change the prompt is through the structured fields.

## 10. Open questions for Amber

1. **persona_tone enum values** — I proposed `formal_sie` / `friendly_du` / `professional_neutral`. Are these the right three? Should `formal_sie` cover both formal **and** professional-neutral (collapsing to two)?
2. **FAQ length caps** — 20 entries × (200 q + 500 a) = 14k chars max contribution to the composed prompt. Acceptable, or tighter?
3. **`appointment_scope` warning UI** — if a customer tries to restrict (e.g. "Nur Wartungstermine"), should the form actively block it with an error, show a warning ("Achtung: Reparaturanrufe werden dann NICHT direkt verbucht"), or just accept? My instinct is **warning, not block** — customers genuinely may want to scope down, but we want them to do it consciously.
4. **Master-template versioning** — when we change the template (e.g. add a new section), composed prompts for all orgs change on next compose call. Should we (a) version the template (`template_v1`, `template_v2`) and let customers opt in to upgrades, or (b) auto-upgrade everyone on deploy with a snapshot taken first? Simpler: **(b)** with a release-note banner on next login.
5. **`uses_composed_prompt` migration default** — I set `default false` so existing orgs (just `kiki-test-007`) stay on legacy. ## 11. Rollout & deferral risks

**Risk: composer output diverges from the hand-tuned raw prompt during migration.** Mitigation: the migration view's side-by-side + super-admin spends time tuning until output matches intent. Once flipped, the safety layer's snapshot is the rollback safety net.

**Risk: customer asks for a field we didn't model.** Either (a) it fits in `special_instructions` (the escape valve), or (b) we add a structured field in a follow-up — but **never** by giving the customer raw-prompt access.

**Risk: master template change accidentally drops a system-managed section.** Mitigation: a unit test on `compose_prompt` that asserts the output contains every `{{system__*}}` placeholder + the booking flow's required verbatim phrases + the booking sequence anchors. Refuse to ship template edits that fail this.

**Risk: super-admin uses raw editor, customer's structured form goes stale.** Mitigation: as noted in §6, any raw write flips `uses_composed_prompt=false` and the customer's editor shows the legacy/migration banner until re-migrated.
