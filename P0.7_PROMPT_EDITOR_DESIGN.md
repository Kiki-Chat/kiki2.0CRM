## 1. Goals & non-goals

**Goals**
1. Customers edit **only** the fields they own. System flow (tool calls, identify-customer scaffolding, booking sequence, system-variable placeholders, webhook URLs) is invisible and unbreakable.
2. The exact regression that triggered this P0 — a customer-editable rule (`CATEGORY CHECK: book only maintenance`) buried in a 50k-char system prompt blocking *all* repair bookings — is **structurally impossible**: no form field permits flow-control logic, and the composer's escape pass neutralizes attempted directive injection (§4, §8.2 Q7 expansion).
3. The agent's final prompt is **deterministically composed** from a versioned master template + the customer's structured fields. Same inputs → same prompt.
4. **Super-admin escape hatch:** the raw prompt + master-template view stays available to super-admins for debugging, with Verlauf disambiguating structured vs. raw edits visually (§6).
5. **Migration:** existing customers (today only `kiki-test-007` is real) move to the new model cleanly via a super-admin migration view, opt-in per org (§5).

**Non-goals**
- Free-form prompt editing for customers (super-admin only).
- Per-section rollback (Verlauf & Rückgängig already covers prompt-level rollback via snapshots).
- A WYSIWYG / no-code prompt builder beyond the field set in §3.
- Multi-language prompts. Per Amber: **DE-formal across the board for all real customers**.
- Renaming or restructuring tools / `client_events` — those are the safety layer's domain, not the editor's.
- **A `persona_tone` enum.** Resolved Q1: tone is contextual and already implicit in the free-text fields (first_message, welcome_message, special_instructions) customers already control. Adding a separate enum forces a binary on something fluid. Dropped from this design — no column, no composer logic, no UI section.

| Field | Why system-managed | How protected |
|---|---|---|
| **Master composition template** | Determines flow, tool call sequence, identify/cancel/book scaffolding. ## 4. Composition logic

### 4.1 Master template (lives in `backend/app/services/prompt_composer.py`) + versioning

Hybrid versioning (resolved Q4): templates live in code as named constants — `TEMPLATE_V1`, `TEMPLATE_V2`, etc. — selected at compose-time by `agent_configs.template_version text default 'v1'`. On deploy of a new template, the composer auto-upgrades on next call (NOT silently — see §6 + §10 Q4) and the old composed prompt is preserved by the safety layer's pre-write snapshot.

`compose_prompt(org_id) -> str`:
1. Loads org row, agent_configs row, business_hours, categories, FAQ (if `faq_enabled`), address, management, `template_version`.
2. Selects the template constant matching `template_version` (falls back to latest if column missing or invalid).
3. Renders each field through an **escape pass** (§8.2 + resolved Q7) that strips `{{...}}`, `{system__*}`, leading `MANDATORY` / `MUST` / `NEVER` / `ALWAYS` / `ONLY` directives, AND markdown headers in a deny-list (`# TOOLS`, `# CATEGORY CHECK`, `# IDENTIFY-CUSTOMER FLOW`, etc.). Customer typing `"MANDATORY: always greet customers warmly"` becomes `"Always greet customers warmly"` — content preserved, system-level authority claim neutered.
4. String-replaces placeholders in the master template. FAQ block is conditional on `faq_enabled` (Path A) or absent entirely (Path B).
5. Returns the final string. Deterministic — same inputs always produce identical output.

**Sketch of the template** (abridged — full version in code):

```text
Du bist Kiki, die Telefon-Assistentin von {{brand_name}}. {{trade_specialty}}

{{welcome_message_if_set}}

Du nimmst Anrufe für {{brand_name}} entgegen. […system flow…]
Die aktuelle Zeit in Deutschland ist {{system__time}}.

### Erkennung & Verarbeitung
[… SYSTEM-MANAGED: Time-of-Day Mode, Existing Customer Reference,
   Step 0, Step 1, Step 2, Step 3 booking flow including the new
   universal scope …]

### Schritt 3 — Termin
{{appointment_scope}}

Der unten beschriebene Buchungsablauf gilt für alle Anliegen.
[… SYSTEM-MANAGED: ask verbatim, narrow down, two suggestions, etc. …]

### Wissensbasis
**Unternehmen**
{{brand_name}} — {{trade_specialty}}.
{{full_address}}.
{{management_block}}

**Geschäftszeiten**
{{business_hours_block}}
Mittagspause: {{lunch_break}}

**Leistungsangebot**
{{services_offered}}

**Leistungen, die wir nicht anbieten**
{{services_not_offered_block}}

{% if faq_enabled %}            ← Path A only; Path B: this whole block absent
**Häufige Fragen**
{{faq_block}}
{% endif %}

**Zusätzliche Hinweise**
{{knowledge_text}}

{{special_instructions}}

### Notfall
[… SYSTEM-MANAGED + emergency_surcharge_text injection …]
```

(In actual Python the conditional is a plain `if faq_enabled: ...` around the slice, not a Jinja template — keeps the dep tree small.)

## 5. Migration plan for existing customers

Unchanged from v1. Briefly:
1. Each org gets `agent_configs.uses_composed_prompt boolean default false` (resolved Q5).
2. Super-admin opens `/super-admin/orgs/{id}/prompt-migration` (a new 3-pane view: current raw prompt left, composer output right, form bottom).
3. Tunes structured fields until output matches intent.
4. `POST /api/super-admin/orgs/{id}/prompt-migration` saves fields + flips `uses_composed_prompt=true` + composes + calls `patch_agent_safely(endpoint_label="prompt-migration")`.
5. From then on, org_admin sees the structured editor; raw editor moves behind `super_admin`-only gate.

**No customer is auto-flipped.** Existing orgs (today only `kiki-test-007`) stay on legacy raw editor until super-admin explicitly migrates them. New orgs created post-deploy also start with `uses_composed_prompt=false`; super-admin flips after structured fields are filled (revisit defaulting to `true` after Phase D — resolved Q5).

**Dependency on P0.6:** the migration view's super-admin route tree depends on P0.6 landing first. Migration view ships as a **follow-up PR after P0.6** (resolved Q6) — same P0.7-build umbrella but separate PR.

## 6. Super-admin escape hatch

The new structured editor REPLACES the raw textarea for `org_admin`. For `super_admin`:
- A toggle `"⚡ Erweiterter Modus"` in the editor toolbar reveals:
  - The existing raw `prompt.prompt` (read-only by default, edit unlocked via a confirmation modal).
  - A `"Master-Template ansehen"` link showing the composer template + field substitution map.
- **Verlauf with visual disambiguation** (resolved Q8):
  - 📋 **"Strukturierte Bearbeitung"** for entries with `endpoint_label="persona-fields"` (structured-form writes).
  - ⚙ **"Erweiterte Bearbeitung"** for entries with `endpoint_label="prompt-editor"` (legacy/super-admin raw writes) or `"prompt-migration"` (super-admin migration view writes).
  - Both fully rollback-able regardless of icon.
- Direct raw edits by super-admin set `agent_configs.uses_composed_prompt = FALSE` until super-admin re-migrates — so the customer doesn't see a stale structured form that no longer matches the deployed prompt.
- Template version upgrades (Q4 hybrid): when a template change is deployed, super-admin sees a banner in the editor `"Template-Version v2 verfügbar — Anwenden?"`. Auto-applying on next compose-call is fine, but the snapshot taken pre-write is the rollback safety net.

## 7. Preview flow + concurrent-write handling

**Preview** (unchanged from v1):
- Sticky bottom-bar `"Vorschau"` button → `POST /api/kiki-zentrale/persona-fields/preview` returns the composed prompt as **read-only string** in a modal.
- Customer cannot paste anything from the preview back into a raw input. Only the structured fields are writable.

**Concurrent write handling** (resolved Q9):
- If a super-admin write is in flight (snapshot row in `agent_config_snapshots` written in the last 10s with no matching audit-success row yet) and an org_admin hits "Speichern":
  - Reject with `409` and message `"Ein Administrator nimmt gerade Änderungen vor — bitte einen Moment warten."`
  - Frontend **auto-refetches** persona-fields state on next window-focus (or via a Realtime broadcast on `org:{org_id}:agent_writes` if/when wired). The org_admin sees the super-admin's changes flow into the form **without** losing their own unsaved edits — the form differentiates "your unsaved value" vs. "server value just changed" with a small chip per field, and the user re-applies their edits intentionally on top of the new baseline.
  - **No automatic merge.** The user is the merger — we just keep them informed.

## 12. Rollout & deferral risks

**Risk: composer output diverges from the hand-tuned raw prompt during migration.** Mitigation: super-admin migration view shows side-by-side + spends time tuning until output matches intent. Safety layer's snapshot is the rollback safety net.

**Risk: customer asks for a field we didn't model.** Either (a) it fits in `special_instructions` (the escape valve), or (b) we add a structured field in a follow-up — but **never** by giving the customer raw-prompt access.

**Risk: master template change accidentally drops a system-managed section.** Mitigation: unit test on `compose_prompt` asserts the output contains every `{{system__*}}` placeholder + the booking flow's required verbatim phrases + the booking sequence anchors. Refuse to ship template edits that fail this. The `template_version` column means we can keep `v1` available as a rollback target while introducing `v2`.

**Risk: super-admin uses raw editor, customer's structured form goes stale.** Mitigation: any raw write flips `uses_composed_prompt=false` and the customer's editor shows the legacy/migration banner until re-migrated.

**Risk: escape pass over-strips legitimate customer content** (e.g., a service-area description that legitimately contains the word "ONLY"). Mitigation: only strip leading-of-line directive keywords + content following them on that line is preserved. Unit tests with adversarial fixtures.
